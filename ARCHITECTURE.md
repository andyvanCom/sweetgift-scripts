# SweetGift Architecture

Документ описывает общую архитектуру платформы SweetGift.ru.

Последнее обновление: 2026-07-10

---

# 1. Назначение проекта

SweetGift развивается как собственная технологическая платформа поверх сайта на Tilda.

Tilda используется как слой отображения и оформления заказа.

Основная логика постепенно переносится в собственную инфраструктуру:

- каталог товаров;
- аналитика;
- рейтинги;
- рекомендации;
- SEO-перелинковка;
- обработка статей;
- фоновые задачи;
- ежедневный контроль состояния системы.

В качестве серверной платформы используется Supabase.

---

# 3. Архитектурные принципы

## 3.1. Модульность

Каждая функция сайта реализуется отдельным JavaScript-модулем.

Модуль можно:

* подключить;
* отключить;
* обновить;
* ограничить определёнными типами страниц.

Изменение одного модуля не должно ломать остальные части системы.

---

## 3.2. Единый загрузчик

Все frontend-модули подключаются через:

```text
sweetgift-loader.js
```

Loader:

* загружает manifest;
* определяет текущий тип страницы;
* подключает необходимые модули;
* не загружает лишний JavaScript;
* контролирует версии файлов;
* предотвращает повторное подключение модулей.

---

## 3.3. Единый Core

Общие функции вынесены в:

```text
sweetgift-core.js
```

Остальные модули используют Core вместо дублирования одинаковой логики.

Core отвечает за:

* RPC-запросы;
* нормализацию URL;
* экранирование HTML;
* работу с cookies;
* работу с localStorage;
* определение типа страницы;
* базовые служебные функции;
* безопасное взаимодействие с backend.

---

## 3.4. Предварительный расчёт данных

Тяжёлые вычисления выполняются заранее.

Примеры:

* рейтинги товаров;
* рекомендации статей;
* рекомендации товаров для статей;
* SEO-блоки карточек товаров;
* агрегированная статистика;
* автоматическая классификация статей.

Frontend получает уже подготовленные данные и только отображает их.

Это снижает:

* нагрузку на браузер;
* количество запросов;
* время загрузки страниц;
* нагрузку на базу данных.

---

## 3.5. Supabase как основной источник данных

Tilda остаётся источником части исходных данных и frontend-слоем.

Supabase используется как единая серверная платформа для:

* хранения каталога;
* хранения статей;
* аналитики;
* рекомендаций;
* автоматизации;
* расчёта рейтингов;
* SEO-связей;
* выполнения фоновых задач.

---

## 3.6. Публичный репозиторий

Репозиторий проекта является публичным.

В репозитории запрещено хранить:

* пароли;
* секретные ключи;
* SMTP-пароли;
* service role key;
* приватные токены;
* внутренние авторизационные значения;
* персональные данные клиентов.

Секретные значения хранятся только в переменных окружения Supabase или в предназначенном для этого хранилище секретов.

---

# 4. Frontend Architecture

## 4.1. Loader

Файл:

```text
sweetgift-loader.js
```

Задачи:

* загрузка manifest;
* определение страницы;
* загрузка Core;
* загрузка зависимых модулей;
* контроль повторной загрузки;
* передача состояния в `window.SG`.

Loader должен подключаться на страницах сайта один раз.

---

## 4.2. Manifest

Файл:

```text
sweetgift-manifest.json
```

Manifest содержит:

* имя модуля;
* имя файла;
* версию;
* статус;
* правила подключения;
* типы страниц.

Пример логики:

```json
{
  "name": "article-stats",
  "file": "sweetgift-article-stats.js",
  "version": "1.0.0",
  "enabled": true,
  "pages": ["/stati/*"]
}
```

После изменения JS-модуля необходимо увеличить его версию.

Это позволяет CDN и браузеру загрузить актуальный файл.

---

## 4.3. Глобальный объект SG

Frontend-модули используют единое пространство имён:

```javascript
window.SG
```

Пример структуры:

```javascript
window.SG = {
  core: {},
  loader: {},
  share: {},
  productAnalytics: {},
  productBadges: {},
  articleStats: {},
  topWidgets: {}
}
```

Использование единого объекта снижает риск конфликтов между скриптами.

Данные используются для:

# 7. Каталог товаров

## 7.1. Источник каталога

Каталог импортируется из товарного фида Tilda.

Импорт выполняется автоматически.

---

## 7.2. Products Catalog

Основная таблица каталога содержит:

* ключ товара;
* название;
* описание;
* цену;
* старую цену;
* изображения;
* URL;
* категорию;
* наличие;
* состав;
* дополнительные свойства.

URL товаров хранятся в нормализованном виде:

* без `editionuid`;
* без UTM-параметров;
* без рекламных параметров;
* без служебных query-параметров.

---

## 7.3. Product Ingredients

Таблица используется для структурирования состава товаров.

Связь:

```text
Товар
  ↓
Ингредиент
  ↓
Категория ингредиента
```

Примеры:

* сыр;
* шоколад;
* клубника;
* кофе;
* орехи;
* мясные деликатесы;
* фрукты.

Данные применяются для:

* фильтров;
* рекомендаций;
* похожих товаров;
* SEO;
* аналитики ассортимента.

---

## 7.4. Product SEO Entities

Для товаров хранятся тематические признаки.

Примеры типов сущностей:

* получатель;
* повод;
* тип товара;
* стиль;
* возраст;
* ценовой сегмент;
* ингредиент.

Используются для автоматической перелинковки и рекомендаций.

---

# 8. Статьи

Статьи являются отдельной подсистемой платформы.

Источник статей периодически импортируется в базу.

---

## 8.1. Articles Index

Таблица содержит индекс статей:

* ключ статьи;
* название;
* URL;
* описание;
* изображение;
* статус;
* SEO-тему;
* служебные поля.

URL статей хранятся без query-параметров.

---

## 8.2. SEO Topics

SEO-темы формируются на основе семантического ядра.

Структура:

```text
Поисковый запрос
    ↓
SEO-группа
    ↓
Статья
```

SEO-темы используются для:

* контроля контент-плана;
* определения закрытых групп запросов;
* поиска отсутствующих статей;
* объединения близких запросов;
* подготовки новых материалов.

---

## 8.3. Article SEO Entities

Для каждой статьи автоматически определяются сущности.

Примеры:

* кому предназначен подарок;
* повод;
* возраст;
* тип подарка;
* стиль;
* ингредиенты;
* формат подарка.

Сущности используются для:

* связи статей с товарами;
* связи товаров со статьями;
* рекомендаций;
* SEO-перелинковки;
* тематических подборок.

---

## 8.4. Article Views

Просмотры статей учитываются отдельно.

Используются:

* общая статистика просмотров;
* суточная статистика;
* популярные статьи;
* анализ сезонности;
* определение материалов для обновления.

Исторические агрегированные данные должны сохраняться длительное время.

---

# 9. Recommendation Engine

Рекомендации строятся на основе заранее рассчитанных данных.

---

## 9.1. Статьи в карточках товаров

Для каждого товара формируется SEO-блок со связанными статьями.

Учитываются:

* получатель;
* повод;
* тип товара;
* стиль;
* возраст;
* другие тематические признаки.

На странице товара выводится готовый набор ссылок.

---

## 9.2. Товары в статьях

Для каждой статьи рассчитывается список подходящих товаров.

Учитываются:

* сущности статьи;
* сущности товара;
* наличие;
* популярность;
* тип товара;
* соответствие теме;
* дополнительные факторы.

Для каждой статьи сохраняется ограниченное количество товаров.

Frontend получает уже готовый список.

---

## 9.3. Популярные товары

Популярность рассчитывается на основе агрегированных событий.

Используется в:

* рейтингах;
* бейджах;
* рекомендациях;
* публичных подборках;
* виджетах.

---

## 9.4. Разнообразие рекомендаций

При развитии алгоритма необходимо избегать однотипных подборок.

Пример:

* клубника в шоколаде;
* букет из клубники;
* подарочная корзина;
* фруктовая корзина;
* подарочный набор.

Цель — показать пользователю разные подходящие форматы подарка.

---

# 10. Analytics

## 10.1. Product Events

Содержит обезличенные события пользователей.

Эта таблица является быстрорастущей.

В дальнейшем для неё может использоваться политика хранения:

* сырые события за ограниченный период;
* агрегированная статистика без ограничения;
* периодическая очистка старых событий.

---

## 10.2. Product Stats

Содержит агрегированные показатели товаров.

Используется для:

* рейтингов;
* карточек;
* бейджей;
* рекомендаций;
* аналитики.

---

## 10.3. Orders Analytics

Подсистема получает событие `tildaform:aftersuccess` от формы корзины Tilda,
снимает состояние корзины до её очистки и вызывает закрытую RPC
`track_product_order`. Повторная отправка одного заказа идемпотентна.

Не должны храниться:

* имя заказчика;
* телефон заказчика;
* имя получателя;
* телефон получателя;
* текст поздравления.

Допустимо хранить обезличенные характеристики:

* состав корзины;
* количество товаров;
* совместно заказанные товары;
* тип доставки;
* заказ себе или в подарок;
* предполагаемый пол заказчика;
* предполагаемый пол получателя;
* тип бесплатной открытки;
* наличие текста поздравления;
* длина текста;
* дата и время заказа.

Frontend передаёт URL страницы и referrer только как pathname, без query-параметров.
Таблица заказов недоступна ролям `anon` и `authenticated` напрямую.

---

## 10.4. Бесплатные открытки

Для аналитики может сохраняться только тип открытки.

Примеры:

* С Днём рождения;
* Спасибо;
* С любовью;
* Поздравляю;
* С 8 Марта;
* С Новым годом;
* Другая.

Сам текст поздравления не сохраняется.

---

# 11. Edge Functions

## 11.1. Import Products

Функция импорта каталога:

* получает товарный фид;
* обновляет каталог;
* обновляет ингредиенты;
* обновляет служебную информацию;
* нормализует URL.

---

## 11.2. Import Articles

Функция импорта статей:

* получает индекс статей;
* добавляет новые статьи;
* обновляет существующие статьи;
* сохраняет URL и метаданные.

---

## 11.3. Classify Articles

Функция автоматической классификации статей.

Выполняет:

* загрузку активных статей;
* анализ названия;
* анализ описания;
* определение сущностей;
* обновление автоматической разметки;
* запуск связанных процедур пересчёта.

---

## 11.4. Send Daily Report

Функция формирует ежедневный технический отчёт.

Отчёт может содержать:

* количество товаров;
* количество статей;
* количество сущностей;
* состояние SEO-блоков;
* статистику просмотров;
* количество рекомендаций;
* состояние ночных задач;
* предупреждения.

Отправка выполняется через отдельный почтовый ящик.

Секреты и почтовые данные хранятся только в переменных окружения.

# 12. Night Pipeline

Каждую ночь выполняется последовательное обновление данных.

Текущая логика:

```text
Импорт товаров
    ↓
Импорт статей
    ↓
Классификация статей
    ↓
Пересчёт SEO-блоков товаров
    ↓
Пересчёт товаров для статей
    ↓
Формирование ежедневного отчёта
```

Популярные списки дополнительно обновляются по отдельному расписанию.

---

## 12.1. Требования к ночному циклу

Каждый этап должен быть повторяемым.

Повторный запуск не должен:

* создавать дубли;
* повреждать существующие данные;
* создавать разные URL одного товара;
* очищать вручную назначенные сущности;
* нарушать работу frontend.

---

## 12.2. Отказоустойчивость

Часть операций может запускаться как внутри основной функции, так и отдельными cron-задачами.

Это допускается временно для повышения надёжности.

После накопления статистики выполнения дублирующие задачи могут быть оптимизированы.

# 13. Daily Report

Ежедневный отчёт используется для быстрого контроля системы.

Пример структуры:

```text
Товары
Статьи
SEO
Перелинковка
Сущности
Аналитика
Заказы
Предупреждения
```

В дальнейшем отчёт должен включать:

* ошибки cron;
* ошибки Edge Functions;
* отсутствие данных;
* пустые SEO-блоки;
* незагрузившиеся frontend-модули;
* аномальное снижение событий;
* аномальный рост ошибок.

# 14. Frontend Health Monitoring

Планируется подсистема контроля frontend-модулей.

Каждый критичный модуль сможет фиксировать события:

* loaded;
* rendered;
* error.

Пример:

```text
core loaded
product-badges loaded
article-stats rendered
top-widget rendered
article-products error
```

Данные должны храниться ограниченное время.

После формирования отчёта старые технические события могут удаляться автоматически.

# 15. Data Retention

Для разных типов данных используется разная политика хранения.

## Долгосрочно

Следует сохранять:

* каталог;
* статьи;
* агрегированную статистику;
* просмотры по дням;
* данные заказов;
* связи товаров;
* рекомендации;
* SEO-сущности.

## Ограниченный срок

Можно хранить ограниченное время:

* сырые события товаров;
* технические frontend-события;
* HTTP-ответы фоновых задач;
* cron-логи;
* временные очереди;
* диагностические данные.

Периоды хранения должны определяться после накопления статистики роста базы.

# 16. URL Normalization

Перед записью URL в базу выполняется нормализация.

Удаляются:

* `editionuid`;
* `utm_source`;
* `utm_medium`;
* `utm_campaign`;
* `utm_content`;
* `utm_term`;
* `yclid`;
* `ysclid`;
* `gclid`;
* `fbclid`;
* другие служебные параметры.

Основной идентификатор товара не должен зависеть от рекламных параметров URL.

# 17. Versioning and Deployment

## 17.1. JavaScript

После изменения JS-файла:

1. увеличить версию в manifest;
2. проверить синтаксис;
3. создать commit;
4. выполнить push;
5. проверить загрузку актуальной версии на сайте.

---

## 17.2. Edge Functions

После изменения Edge Function:

1. проверить файл локально;
2. выполнить deploy;
3. выполнить тестовый запуск;
4. проверить ответ;
5. проверить изменения в базе;
6. зафиксировать изменение в CHANGELOG.

---

## 17.3. Database

Изменения базы должны фиксироваться:

* SQL-миграцией;
* записью в DATABASE.md;
* записью в CHANGELOG.md.

Не рекомендуется выполнять важные изменения только вручную без сохранения SQL.

# 21. Long-Term Goal

SweetGift должен стать автономной интеллектуальной платформой.

* источник данных;
* аналитическая платформа;
* Recommendation Engine;
* SEO Engine;
* система автоматизации;
* инфраструктура фоновых задач.

Цель архитектуры — обеспечить развитие проекта без зависимости от ограничений стандартного функционала Tilda.
