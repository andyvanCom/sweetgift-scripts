# SweetGift Database

Документ описывает логическую структуру базы данных платформы SweetGift.

Последнее обновление: 2026-07-10

---

# Назначение

Supabase является основной серверной платформой проекта.

База данных используется для хранения:

- каталога товаров;
- статей;
- SEO-структуры;
- рекомендаций;
- агрегированной аналитики;
- технической информации;
- результатов автоматической обработки данных.

Основной принцип архитектуры — выполнять ресурсоёмкие вычисления заранее и использовать готовые данные на frontend.

# Основные подсистемы

## Каталог товаров

Подсистема хранит информацию обо всех товарах интернет-магазина.

Основные возможности:

- автоматический импорт каталога;
- нормализация данных;
- классификация товаров;
- хранение состава;
- подготовка данных для рекомендаций;
- аналитика ассортимента.

---

## Статьи

Статьи являются самостоятельной подсистемой.

Для каждой статьи автоматически определяются:

- SEO-тема;
- тематические сущности;
- связи с товарами;
- статистика просмотров.

Это позволяет автоматически связывать информационный контент с каталогом.

---

## SEO Engine

SEO Engine отвечает за автоматическое определение тематических связей.

Используется для:

- автоматической перелинковки;
- тематических подборок;
- поиска отсутствующих материалов;
- формирования рекомендаций;
- анализа покрытия семантики.

Все вычисления выполняются заранее.

---

## Recommendation Engine

Подсистема рекомендаций объединяет данные каталога, статей и аналитики.

Используется для:

- статей внутри карточек товаров;
- товаров внутри статей;
- популярных подборок;
- рейтингов;
- тематических блоков.

Frontend получает только заранее рассчитанные рекомендации.

---

## Analytics

Подсистема аналитики собирает обезличенные действия пользователей.

Используется для:

- популярности товаров;
- построения рейтингов;
- анализа поведения пользователей;
- формирования рекомендаций;
- развития алгоритмов платформы.

---

## Orders Analytics

Планируется отдельная подсистема аналитики заказов.

В базе должны храниться только обезличенные характеристики заказа.

Персональные данные клиентов не используются в аналитических алгоритмах.

---

# Backend

Используется:

- PostgreSQL;
- Edge Functions;
- Scheduled Jobs;
- агрегированные таблицы;
- серверные процедуры.

Все вычисления выполняются на стороне backend.

Frontend не должен выполнять сложные SQL-запросы.

# Политика хранения данных

## Долгосрочное хранение

Без ограничения срока рекомендуется хранить:

- каталог товаров;
- статьи;
- агрегированную статистику;
- рекомендации;
- SEO-сущности;
- ежедневую статистику просмотров.

---

## Ограниченный срок хранения

Со временем может очищаться:

- сырая аналитика;
- технические журналы;
- диагностические записи;
- временные данные.

Политика хранения определяется отдельно для каждой подсистемы.

---

# URL Normalization

Перед сохранением ссылки автоматически нормализуются.

Удаляются:

- рекламные параметры;
- служебные query-параметры;
- временные идентификаторы;
- параметры кампаний.

Это позволяет избежать появления нескольких записей одного и того же объекта.

---

# Правила развития базы данных

При добавлении новой подсистемы необходимо:

- использовать нормализованные ключи;
- избегать дублирования данных;
- выполнять тяжёлые вычисления заранее;
- документировать изменения;
- обновлять CHANGELOG;
- использовать агрегированные данные вместо повторных вычислений.

---

# Безопасность

В базе данных не должны храниться:

- SMTP-пароли;
- токены доступа;
- секретные ключи;
- конфиденциальные настройки.

Персональные данные используются только там, где это необходимо для работы магазина, и не применяются в аналитических механизмах.

---

# Долгосрочная цель

База данных постепенно становится единым источником знаний платформы.

Все подсистемы — каталог, статьи, SEO, рекомендации, аналитика и будущие AI-функции — используют единую структуру данных.

Frontend отвечает только за отображение информации, а вся бизнес-логика и вычисления выполняются на стороне платформы.
