# SweetGift Functions

Документ описывает серверные функции, RPC-процедуры и плановые задачи платформы SweetGift.

Последнее обновление: 2026-07-10

---

# Назначение

FUNCTIONS.md является техническим справочником автоматических процессов платформы.

Документ содержит:

- Edge Functions;
- RPC-функции;
- плановые задачи;
- последовательность ночной обработки;
- назначение основных процедур;
- зависимости между функциями.

Документ не содержит:

- секретов;
- токенов;
- паролей;
- закрытых URL;
- service role key;
- SMTP-настроек;
- внутренней авторизационной информации.

---

# Общая схема

```text
Источники данных
      │
      ▼
Edge Functions
      │
      ▼
PostgreSQL
      │
      ├── RPC
      ├── Aggregation
      ├── SEO
      └── Recommendations
      │
      ▼
Frontend Modules
```

# Edge Functions

## import-yml-products

### Назначение

Импорт каталога товаров из внешнего товарного фида.

### Выполняет

- загрузку товарного фида;
- разбор товаров;
- обновление каталога;
- обновление состава;
- обновление изображений;
- обновление цен;
- обновление статусов наличия;
- нормализацию URL;
- обновление информации об источнике импорта.

### Основные таблицы

- products_catalog;
- product_ingredients;
- feed_sources.

### Особенности

Повторный запуск не должен создавать дубли товаров.

Один товар должен соответствовать одной записи по нормализованному ключу.

---

## import-articles-index

### Назначение

Импорт и обновление индекса статей.

### Выполняет

- загрузку списка статей;
- добавление новых материалов;
- обновление существующих материалов;
- сохранение заголовков;
- сохранение URL;
- сохранение описаний;
- сохранение изображений;
- обновление статуса статьи.

### Основная таблица

- articles_index.

### Особенности

URL статей должны сохраняться без рекламных и служебных параметров.

---

## classify-articles

### Назначение

Автоматическая тематическая классификация статей.

### Выполняет

- получает список активных статей;
- анализирует заголовок;
- анализирует описание;
- анализирует URL;
- определяет получателя подарка;
- определяет повод;
- определяет возраст;
- определяет тип товара;
- определяет стиль;
- определяет ингредиенты;
- удаляет старые автоматические сущности;
- записывает новую автоматическую разметку;
- запускает связанные процедуры пересчёта.

### Основные таблицы

- articles_index;
- article_seo_entities.

### Типы сущностей

- recipient;
- occasion;
- age;
- product_type;
- style;
- ingredient.

### Источник сущностей

Автоматические сущности сохраняются с отдельным признаком источника.

Ручная разметка не должна удаляться при повторной классификации.

---

## send-daily-report

### Назначение

Формирование и отправка ежедневного технического отчёта.

### Может включать

- количество товаров;
- количество активных товаров;
- количество статей;
- статьи без SEO-темы;
- статьи без сущностей;
- товары без сущностей;
- состояние SEO-блоков;
- количество рекомендаций;
- статистику просмотров;
- количество заказов;
- предупреждения;
- результаты ночной обработки.

### Особенности

Почтовые настройки и секреты хранятся только в переменных окружения.

# RPC Functions

## track_product_event

### Назначение

### Используется

Frontend-модулем аналитики товаров.

### Примеры событий

- view;

### Результат

Событие сохраняется для последующей агрегации.

---

## refresh_all_top_lists

### Назначение

- периоды;

### Назначение

Поддерживаемые периоды могут включать:

- сегодня;
- вчера;
- неделю;
- месяц;
- квартал.

---

## get_public_top_lists_page_period

### Назначение

Получение готовых популярных подборок за выбранный период.

### Используется

- Top Pages;
- Top Widgets;
- публичными рейтингами;
- тематическими блоками.

### Результат

Возвращает подготовленный список товаров без дополнительных вычислений на frontend.

---

## refresh_product_card_seo_blocks_all

### Назначение

Массовая пересборка SEO-блоков карточек товаров.

### Выполняет

- анализ сущностей товара;
- поиск подходящих статей;
- сортировку по релевантности;
- ограничение количества ссылок;
- сохранение готового блока.

### Основная таблица

- product_card_seo_blocks.

### Особенности

Frontend получает уже готовый блок.

---

## refresh_product_card_seo_blocks_one

### Назначение

Пересборка SEO-блока для одного товара.

Используется для:

- тестирования;
- точечного обновления;
- ручной проверки;
- локального исправления.

---

## refresh_article_product_recommendations

### Назначение

Пересчёт товаров, рекомендованных для статей.

### Выполняет

- сопоставление сущностей статьи и товара;
- проверку доступности;
- расчёт релевантности;
- сортировку;
- присвоение rank;
- сохранение причины выбора;
- ограничение количества товаров.

### Основная таблица

- article_product_recommendations.

### Стандартный лимит

До 8 товаров на статью.

# Scheduled Jobs

Плановые задачи выполняются средствами PostgreSQL и Supabase.

---

## import-yml-products-daily

### Назначение

Ежедневное обновление товарного каталога.

### Выполняется

Один раз в сутки перед остальными этапами ночной обработки.

Дополнительно сверяет каталог с полным YML и помечает отсутствующие в источнике
товары недоступными.

---

## import-articles-index-daily

### Назначение

Ежедневное обновление индекса статей.

### Выполняется

После импорта товаров и до классификации статей.

Daily-режим обрабатывает весь список новых и изменённых статей за один запуск,
а отсутствующие в sitemap статьи помечает неактивными.

---

## classify-articles-daily

### Назначение

Ежедневная автоматическая классификация статей.

### Выполняется

После импорта статей.

### Может дополнительно запускать

- пересборку SEO-блоков;
- пересчёт рекомендаций;
- обновление рейтингов.

---

## refresh-product-card-seo-blocks-daily

### Назначение

Гарантированная отдельная пересборка SEO-блоков карточек товаров.

### Причина отдельного запуска

Используется как резервный этап ночной обработки.

---

## refresh-article-product-recommendations-daily

### Назначение

Гарантированный отдельный пересчёт товаров для статей.

### Причина отдельного запуска

Используется как резервный этап, даже если пересчёт уже запускался внутри классификатора.

---

## refresh-top-lists-hourly

### Назначение

Почасовое обновление популярных товаров и рейтингов.

### Используется

Для поддержания актуальности:

- Top Pages;
- Top Widgets;
- Product Badges;
- рейтингов товаров.

---

## send-daily-report

### Назначение

Отправка утреннего технического отчёта.

### Выполняется

После завершения ночного pipeline.

Письмо считается успешным только если обязательные задачи свежие и успешные,
а количества товаров и статей в источниках совпадают с активными записями базы.

# Night Pipeline

Текущая последовательность:

```text
Импорт товаров
      ↓
Импорт статей
      ↓
Классификация статей
      ↓
Пересчёт SEO-блоков товаров
      ↓
Пересчёт товаров для статей
      ↓
Ежедневный отчёт
```

Популярные списки дополнительно обновляются каждый час.

# Надёжность

Некоторые процедуры могут запускаться дважды:

- внутри Edge Function;
- отдельной cron-задачей.

Это временно допустимо для повышения надёжности.

Повторный запуск должен быть идемпотентным и не создавать дубли.

---

# Требования к функциям

Каждая функция должна:

- безопасно обрабатывать повторный запуск;
- не создавать дубли;
- возвращать понятный статус;
- сообщать количество обработанных записей;
- фиксировать ошибки;
- не раскрывать секреты;
- не хранить персональные данные без необходимости.

---

# Обработка ошибок

Ошибки должны:

- возвращаться в структурированном виде;
- попадать в технический отчёт;
- не оставлять данные в частично обновлённом состоянии;
- позволять повторный запуск задачи.

Ошибки фоновых задач записываются в отдельный журнал выполнения.

---

# Логирование

Используется таблица:

system_job_logs

- имя задачи;

- время завершения;
- статус;
- количество обработанных записей;
- длительность;

# Order Analytics

## track_product_order

Принимает JSON успешно оформленного заказа, проверяет обязательный `order_id` и
от 1 до 100 позиций, затем идемпотентно записывает их в `product_orders`. Событие
`purchase` создаётся только для действительно добавленной позиции, поэтому повтор
запроса не искажает статистику.

Доступна ролям `anon` и `authenticated`; сама таблица им недоступна.
